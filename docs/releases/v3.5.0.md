# Konoha Agentic Academy v3.5.0 — Conversational Hokage Product

## Summary

v3.5.0 turns the canonical `konoha` entrypoint into a conversational,
local-first and explicitly supervised mission product.

The operator can describe a repository mission in natural language and remain
inside `Mission>` while Hokage coordinates intention capture, Mission Charter
approval, bounded actions, deterministic validation, local-model evidence,
exact patch approval, tests, human review, Teachback, closure and private
memory.

## Product flow

```text
natural-language request
→ Mission Charter
→ explicit Charter approval
→ deterministic checks
→ one-use local Ollama grant
→ repository audit
→ suggested / validated / suppressed findings
→ exact patch preview and SHA-256
→ separate patch approval
→ controlled apply
→ post-patch tests
→ human review
→ human Teachback
→ explicit closure
→ private Obsidian-compatible memory
→ returning-session continuity
```

## Main capabilities

- Bare `konoha` opens the Conversational Hokage.
- Intent and Mission Charter artifacts are deterministic and persisted.
- Action approvals are bound to immutable argument hashes.
- Local Ollama access uses a mission-scoped, one-use grant.
- The selected local model and exact Ollama host are propagated consistently.
- Model suggestions, validated findings and suppressed false positives remain
  separate.
- Patch proposals include exact operations, changed paths, unified diff and
  SHA-256.
- Only validated findings may enter a patch plan.
- Controlled apply does not authorize Git.
- Post-patch tests precede review and closure.
- Teachback is human-authored and closure remains a separate gate.
- Mission, decision, context-pack and local-model audit notes are written only
  to private ignored memory.

- provider: Ollama;
- model: `qwen2.5-coder:1.5b`;
- input tokens reported by the provider: 2050;
- output tokens reported by the provider: 800;
- four bounded actions completed;
- deterministic pre-model checks: passed with zero failures;
- exact README documentation patch: approved and applied;
- post-patch tests: passed with zero failures;
- human review: approved;
- Teachback: passed;
- mission closure: passed;
- continuity reentry: passed;
- private memory outputs: four.

The model produced evidence only. A deterministic release guard supplied the
single validated issue used for the acceptance patch.

## Safety

- No external model service is enabled.
- Models are not downloaded automatically.
- No arbitrary shell is introduced.
- No daemon, web server or background autonomous agent is introduced.
- Model output is evidence only.
- Memory is evidence only.
- Patch proposals are not permission.
- Git stage, commit, merge, tag, push and GitHub release remain explicit,
  separate human gates.
- Private Kirigakure reports, workspaces and Obsidian memory remain ignored.

## Known limitations

- The real audit flow supports the existing bounded documentation patch
  operations; general arbitrary code generation is out of scope.
- Ollama and the selected model must already be installed locally.
- Provider token counts are retained as evidence but are not used as authority.
- GitHub publication requires the maintainer's existing Git and GitHub CLI
  credentials.
