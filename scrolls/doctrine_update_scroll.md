# Doctrine update scroll

This Scroll defines how doctrine changes are proposed, reviewed, approved, and recorded.

## Core rule

No doctrine changes without explicit approval.

## Status

Draft.

## Activation

Use this Scroll when a mission may modify:

- core laws;
- conduct;
- protocols;
- clan rules;
- Scroll behavior;
- review requirements;
- approval policy;
- memory policy;
- safety rules;
- AGENTS.md;
- public guidance that changes agent behavior.

## Default mode

Doctrine update is planning-only until approval.

No agent may edit doctrine files without an approved Mission Charter that explicitly allows the target files.

## Inputs

A doctrine update proposal must include:

- current rule;
- problem observed;
- evidence;
- proposed change;
- affected files;
- compatibility with existing laws;
- risk of overfitting to one mission;
- local vs public scope;
- review level required.

## Evidence

Acceptable evidence includes:

- repeated mission failures;
- review findings;
- safety incidents;
- approved learning proposals;
- user decisions;
- contradictions between current docs;
- missing rule that caused confusion.

Do not update doctrine because a model "thinks" it is better.

## Drafting

Shikamaru may draft doctrine changes when approved.

The draft must:

- use clear language;
- avoid broad ambiguous authority;
- state forbidden actions;
- state escalation path;
- preserve safety hierarchy;
- include examples only when helpful;
- avoid copying protected material.

## Review

Doctrine changes require at least Jounin review.

Escalate to Kage Summit when the change affects:

- safety;
- approval;
- memory;
- local privacy;
- external publication;
- agent authority;
- learning workflow;
- execution permissions.

## Report

The doctrine update report should include:

```text
Files changed:
Reason:
Evidence:
Risk:
Review:
User approval:
Diff summary:
Follow-up:
```

## Completion

A doctrine update is complete only when the user approves the final wording and the change is traceable in Git.
